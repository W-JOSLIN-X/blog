刚学数据库时，很容易把所有字段堆在一张表里。等项目跑了两个月，发现一个用户有多个地址却只存了一个，或者改一个分类名字要扫全表 update —— 这时候就该回头补范式理论了。

## 范式化：让数据各归其位

**第一范式（1NF）**：每列都是原子值。

```sql
-- 反例：skill 列存了逗号分隔的多个值
INSERT INTO person (name, skill) VALUES ('张三', 'Java,Python,SQL');

-- 正例：拆到另一张关联表
CREATE TABLE person_skill (
    person_id INT,
    skill VARCHAR(50),
    PRIMARY KEY (person_id, skill)
);
```

破坏 1NF 的直接后果：想查 "所有会 Python 的人"，只能用 `LIKE '%Python%'`，索引失效且可能误匹配。

**第二范式（2NF）**：非主键列完全依赖于主键（消除部分依赖）。常见于联合主键场景 —— 如果课程名只取决于课程 ID，那它不该出现在"选课表"里，应该放到"课程表"。

**第三范式（3NF）**：非主键列不传递依赖于主键。

```sql
-- 反例：总价 = 单价 × 数量，是属于字段间的推导关系
CREATE TABLE orders (id INT, price DECIMAL(8,2), qty INT, total DECIMAL(8,2));

-- 正例：total 不存储，查询时计算
SELECT id, price, qty, price * qty AS total FROM orders;
```

## 命名约定

团队内统一规范比拍脑袋重要。个人习惯：
- 表名用复数或单数统一，推荐 `users` / `orders`
- 关联表取名 `a_b`，如 `user_role`
- 主键统一用 `id`，外键用 `关联表_关联字段`，如 `user_id`
- 布尔字段用 `is_active`、`has_paid`，避免 `status` 这种歧义名

## 主键选择

整数自增 ID 简单省空间，够用。UUID 适合分布式环境但索引碎片化严重。雪花算法（Snowflake）是折中方案，时间有序的 Long 型 ID 兼顾趋势递增和分布式。

避免用业务字段做主键 —— 手机号可能回收、身份证号涉及隐私脱敏、邮箱可能修改，这些变数会折腾所有关联表的外键。

## 什么时候该反范式

读多写少、报表查询频繁时，适当冗余可以免去多表 JOIN。比如订单表里冗余存一个 `user_name`，用户改了名字再说。关键是要**有意为之**，并有补偿机制。

## 索引基本原则

- 高频查询的 WHERE 列、JOIN 列、ORDER BY 列考虑加索引
- 高基数列（性别、状态等取值少的字段）不适合单独索引
- 联合索引满足最左前缀原则
- 用 `EXPLAIN` 验证执行计划，不要凭感觉加索引

## 一个常见坑

不要把计算值存成物理列并且直接信以为真。`balance = 100` 这个数字是不是和流水对得上？存计算结果时，至少保留"原始交易记录"作为可追溯的数据源。数据库里的真值应该是不可变的事件记录，而不是可被覆盖的快照。

把建模当成在回答一个问题："如果业务规则变了，我要改多少处代码？" 范式化和命名规范的最终目的，就是让这个答案尽可能接近 "一处"。
