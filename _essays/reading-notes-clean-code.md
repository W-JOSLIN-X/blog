期末考试结束之后，总算有时间静下来读几本一直想看的书。《代码整洁之道》就是其中之一。

翻开这本书之前，我多少有点"洁癖"——变量名要语义明确，缩进要对齐，函数不能太长。但 Martin 还是给了我一记重锤：命名这件事，远比我想象的重要。好的命名可以让你六个月后看自己的代码时不需要骂自己。取名字的时间不应该省，因为你只能在两条路里选一条：花五分钟现在想清楚，或者花五十分钟以后改 bug。

"函数应该只做一件事。"这句话我听过很多遍，但 Martin 把它掰开揉碎地讲了：如果函数里做的事情不在同一层抽象，那它就在做不止一件事。我回头翻了翻自己上学期写的课设代码，很多函数里混杂了校验、计算和格式化输出——以后得改。

关于注释，Martin 的观点很犀利：注释是对表达失败的补偿。能用代码表达清楚的事情，就不要指望注释来补救。真正好的代码，读起来就像在讲故事。

最后是童子军军规——"让营地比你到达时更干净"。每次提交代码，顺手把经过的地方稍微整理一下。这一点点积累，长期下来会让整个代码库保持良好的状态。

这本书不厚，但值得反复翻。等用到项目里再说。
